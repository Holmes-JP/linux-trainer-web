## 「No space left on device」は突然来る

ファイルが保存できない、ログが書けない、アプリが起動しない。その多くの正体が **ディスクフル** だ。

```
$ touch newfile
touch: cannot touch 'newfile': No space left on device
```

このエラーが出たら、慌てて消す前に **どこが、なぜ埋まったか** を順に突き止める。使う道具は「全体を見る `df`」「犯人を探す `du`」「装置を見る `lsblk`」の3つだ。

## df — どの領域が満杯か

`df` は **ファイルシステム単位** の使用量を表示する。まずこれで「そもそもどこが埋まっているのか」を掴む。`-h` で読みやすい単位になる。

```
$ df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/root        20G   20G     0 100% /
tmpfs           1.0G     0  1.0G   0% /dev/shm
```

見るのは **Use%** と **Mounted on** の組み合わせだ。上の例では `/` (ルート) が 100%。**どのディレクトリの下が満杯なのか** がここで分かる。逆に言えば、埋まっている領域と無関係な場所をいくら消しても解決しない。

## du — 犯人ディレクトリを掘り当てる

満杯の領域が分かったら、その中で **容量を食っているディレクトリ** を `du` で探す。いきなり全体を見ると膨大なので、階層を1つずつ下りるのがコツ。

```
$ du -h --max-depth=1 /var
16G     /var/log
40M     /var/cache
16G     /var
```

`--max-depth=1` で「直下の各ディレクトリの合計」だけを出す。大きいものを見つけたら、そのディレクトリを次の `du` の対象にして、また1階層下りる。これを繰り返して犯人 (肥大したログ、巨大な一時ファイルなど) に辿り着く。

大きな **ファイル** を直接探すなら `find` が速い。

```
$ find /var -type f -size +500M
/var/log/app.log
```

## lsblk — 装置としてのディスクを見る

`df` は「使われている領域」を見るが、`lsblk` は **物理的なブロックデバイス** (ディスクとその分割) をツリーで見せる。

```
$ lsblk
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
vda    254:0    0   20G  0 disk
└─vda1 254:1    0   20G  0 part /
```

`MOUNTPOINTS` が空のデバイスは「装置はあるが、まだファイルツリーに繋がっていない」状態だ。ここで `df` に出ない容量の存在に気づくことがある。

## mount / umount の考え方

Linux では、ディスク (デバイス) はそのままでは使えない。**ファイルツリーのどこかのディレクトリに接続** して初めて読み書きできる。この接続が **マウント**、接続先が **マウントポイント** だ。

```
# mount /dev/vdb1 /mnt/data     ← デバイスを /mnt/data に繋ぐ
# umount /mnt/data              ← 接続を外す
```

つまり `/mnt/data` へのアクセスは、実際には `/dev/vdb1` という装置への読み書きになる。`df` の「Mounted on」列は、この対応表を見ているわけだ。マウント操作は装置を触るため root の仕事で、`su root` (パスワード `password`) で root になってから行う。

> この演習環境 (ブラウザ内の CheerpX) は単一のルートで完結しているため、追加のディスクを繋ぐ場面は少ない。ただし「装置とマウントポイントは別物」という考え方は、`df` の読み方そのものなので押さえておきたい。

## トラブル対応の型

1. `df -h` で **どのマウントポイントが 100%** かを特定する (埋まっている領域を確定させる)
2. その領域の中を `du -h --max-depth=1` で上から掘り、容量を食うディレクトリへ下りていく
3. 巨大ファイルそのものを狙うなら `find パス -type f -size +500M` で一発で出す
4. 消してよいと確認できたものだけ削除する (ログや一時ファイルが定番。設定やデータは消さない)
5. 装置レベルで足りないなら `lsblk` で未使用デバイスを確認し、root で `mount` して容量を足す
