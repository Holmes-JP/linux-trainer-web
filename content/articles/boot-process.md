## 電源を入れてから操作できるまで

「電源ボタンを押したら Linux が起動する」の裏では、いくつもの段階がバトンをつなぐように進んでいる。全体の流れを 5 段階でつかんでおくと、起動しない・サービスが上がらないといったトラブルのとき、**どの段階の問題か** を切り分けられるようになる。

まずは大きな流れを一望しよう。

| 段階 | 主役 | やること |
|---|---|---|
| 1 | ファームウェア (BIOS / UEFI) | ハードウェアを初期化し、起動ディスクを探す |
| 2 | ブートローダ (GRUB) | カーネルを選んでメモリに読み込む |
| 3 | カーネル + initramfs | ドライバを読み、ルート FS をマウントする |
| 4 | init システム (PID 1) | サービスを順に立ち上げる |
| 5 | ログイン / シェル | 人が操作できる状態になる |

## 1. ファームウェア (BIOS / UEFI)

電源が入ると、まずマザーボードに焼き込まれた **ファームウェア** が動く。古くからある **BIOS**、現代の **UEFI** がこれにあたる。役割は、CPU やメモリなど **ハードウェアの最低限の初期化** と、動作チェック、そして「どのディスクから起動するか」を決めて次のバトンを渡すことだ。

## 2. ブートローダ (GRUB)

ファームウェアが起動ディスクを見つけると、そこに置かれた **ブートローダ** に制御が移る。Linux で広く使われるのが **GRUB** だ。役割は、起動する **カーネルを選んでメモリに読み込む** こと。カーネルが複数あるとき (新旧のバージョンなど) に選択メニューを出すのも GRUB の仕事だ。

## 3. カーネル起動 + initramfs

読み込まれた **カーネル** が動きはじめる。ここが Linux 本体の起動だ。ただしこの時点では、本物のディスク (ルートファイルシステム) をまだマウントできない。ディスクを読むためのドライバが、そのディスクの中にある、というニワトリと卵の問題があるからだ。

これを解くのが **initramfs** — メモリ上に展開される小さな仮のファイルシステムで、**最低限のドライバ** を積んでいる。カーネルはまず initramfs で起動し、必要なドライバを読み込んで、ようやく本物のルート FS を `/` にマウントする。

起動中にカーネルやドライバが出したメッセージは、後から **`dmesg`** で読める。

```
$ dmesg | head -3
[    0.000000] Linux version 6.1.0 ...
[    0.512000] Memory: 4096MB available
[    1.204000] sda: sda1 sda2
```

「起動時にこのデバイスは認識された？」を確かめたいときの定番が `dmesg` だ。

## 4. init システム (PID 1)

ルート FS がマウントできたら、カーネルは最初のプロセスとして **init システム** を起動する。これが **PID 1** — すべてのプロセスの祖先になる特別なプロセスだ。役割は、必要な **サービス (デーモン) を正しい順序で立ち上げていく** こと。

init システムには歴史的に 2 系統ある。

| 方式 | 立ち位置 | 操作コマンド |
|---|---|---|
| **SysV init** (古典) | 起動スクリプトを 1 本ずつ順に実行 | `service 名前 start` |
| **systemd** (現代の主流) | 依存関係を解決し並列で起動 | `systemctl start 名前` |

現代のほとんどの本番サーバーは **systemd** が PID 1 として動いている。systemd では、起動時にどこまでの状態を立ち上げるかを **ターゲット (target)** という単位で表す。これは **昔の「ランレベル」に相当する** もので、たとえば「ネットワーク付きの通常稼働状態」といったゴールをまとめて指す。

いま自分が動かしているカーネルは `uname -a`、PID 1 に何が居るかは `ps -p 1` で確認できる。

```
$ ps -p 1
  PID TTY          TIME CMD
    1 ?        00:00:01 systemd
```

## 5. ログイン / シェル

init システムがサービスを立ち上げ終えると、**ログイン画面やシェル** が現れ、ようやく人が操作できる状態になる。ここまで来て初めて、私たちがコマンドを打てるようになる。電源投入からここまでが「ブート」だ。

## この演習環境 (WebVM) では systemd が動いていない

ここが実務との大事な違い。**この演習環境 (ブラウザ内の WebVM / CheerpX Debian) は systemd で起動していない**。PID 1 に systemd が居ないため、`systemctl` や `journalctl` は使えず、打つと次のように弾かれる。

```
$ systemctl status ssh
System has not been booted with systemd as init system (PID 1). Can't operate.
```

混乱しないよう、現実のサーバーと本演習を対比で押さえておこう。

| やりたいこと | 現実のサーバー (systemd) | この演習 (WebVM) |
|---|---|---|
| サービスを起動 | `systemctl start 名前` | `service 名前 start` |
| サービスのログを読む | `journalctl -u 名前` | `/var/log/` のファイルを直接読む |
| 起動・デバイスのメッセージ | `dmesg` / `journalctl -k` | `dmesg` |

サービスの起動・停止には root 権限が要るので、`su root` (パスワード: `password`) になってから `service` で操作する。

## 覚え方

- ブートは **ファームウェア → GRUB → カーネル+initramfs → init (PID 1) → シェル** の 5 段リレー
- **PID 1 = init システム** がすべてのサービスの司令塔。現代は systemd、systemd のターゲットは昔のランレベルに相当する
- 本演習は systemd 非搭載 → **サービスは `service`、ログは `/var/log`、起動まわりは `dmesg`** で調べる
